Case Study

1. Introduction

Trellis is an open-source, low-config deployment pipeline for serverless applications. It minimizes the number of steps required for teams to start deploying their code to the cloud, and its dashboard provides a centralized space for teams to monitor and manage multiple deployment environments.

Trellis handles the complexity of setting up an automated deployment pipeline for serverless applications. This makes it suitable for teams who don’t want to worry about allocating engineering resources for setting up and maintaining one themselves. Trellis takes care of the configuration so developers can focus on adding business value by coding new features and testing them out in the cloud.

This case study will shed light on the value of Trellis by addressing the problem it solves and exploring the underlying complexity that hides. It will also describe the various challenges, trade-offs, and choices faced in building Trellis.

2. Technical Background

To set a firm foundation for the objectives of Trellis, we will first examine the technical background of the overarching problem domains: serverless applications and deployments.

2.1 Serverless Applications

2.1.1 What is Serverless

Serverless is an architectural model that frees software developers from the need to consider and manage servers while creating and running applications. Serverless computing allows cloud providers to manage the underlying physical infrastructure, relieving developers of the burden of doing it themselves2-1.

2.1.2 Why Serverless

There are several reasons to choose a serverless application architecture, including 2-2:

  • Lower startup costs: developers can start prototyping high-quality software without needing to make significant infrastructure investments. With the full power of a cloud provider available over the internet, even small teams can build and release enterprise-scale software.
  • No server management necessary means less time, money, and work allocated to a dedicated DevOps team.
  • Developers only pay for the server space they use, reducing resource waste.
  • Applications are automatically scaled as needed, so servers are automatically scaled out and back in according to real-time traffic. This auto-scaling capability is ideal for bursty or otherwise unpredictable traffic patterns.
  • Code can run closer to users as it is not geographically limited to an organization’s physical location.

While serverless architectures offer several perks, they are not necessarily suitable for all teams; some might want to avoid vendor lock-in, manage their physical infrastructure, or have servers up and running constantly. Nonetheless, over half of all organizations operating in the cloud have adopted serverless technologies2-3, indicating that the serverless paradigm has become essential to modern development.

2.1.3 Serverless Development

Developing a serverless application involves choosing a cloud provider, determining which of its managed services are needed, and then stitching them together to provide the application’s required functionality2-4. Cloud providers offer hundreds of services and infrastructure components, and while they aim to make them as accessible as possible, developers still need to do the work of specifying what resources to provision and how to configure them.

As applications grow in size and utilize an increasing number of cloud components, developers face several challenges in manually provisioning and managing infrastructure, such as environment drift. Environment drift occurs when there is a discrepancy between environments that are designed to be the same. To manage these challenges, cloud providers and third parties have utilized a model called Infrastructure as Code. Infrastructure as Code (IaC) is a model that allows developers to provision their infrastructure using code.

With IaC, engineers can document, edit, distribute, and replicate their infrastructure configurations using a familiar programming language2-5. As such, application infrastructure can be coupled with the logic that depends on it, turning source code and its corresponding builds into comprehensive artifacts that define everything the application needs to function [^5]. Because these artifacts precisely codify the infrastructure they rely on, they can be deployed with minimal manual configuration.

2.1.4 Serverless Development With Serverless Stack

There are several cloud providers, each with multiple IaC tooling solutions for developers to choose from. The options include first-party solutions offered by the cloud provider directly (e.g. AWS CDK, GCP Deployment Manager, Axure Bicep), vendor-agnostic tools built on first-party APIs (e.g. Terraform, Pulumi), and even frameworks that abstract away common configurations and implement opinionated approaches to common full-stack development practices (e.g. Serverless Framework, Serverless Stack).

In developing Trellis we have specifically chosen to support applications developed using the Serverless Stack (SST) Framework. SST is a JavaScript framework which extends AWS CDK. In addition to being an IaC tool it offers several useful features for developers of serverless applications. We chose to support SST because it[^6]:

  • Strives to make it easy to build serverless applications on AWS, and AWS is the most-widely-used cloud provider, consuming 30% of the global cloud market
  • Provides high-level, composable constructs designed specifically for serverless applications, but also supports native CDK constructs when needed.
  • Is built on the AWS JavaScript CDK (Cloud Development Kit), which makes it accessible to millions of JavaScript users around the world.
  • Offers development tools like an infrastructure management console and live local lambda development, which enables debugging breakpoints and instant feedback on changes made to lambda functions.

While these features make SST a helpful development framework, developers still need to implement best practices throughout the entire software development life cycle to consistently release quality work. In particular, Trellis builds on SST to help teams follow some of the best practices regarding application deployment.
